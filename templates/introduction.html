About

This is software commissioned by a roofing company that wanted improvements in efficiency to their backoffice invoicing department. It bundles multiple pieces of integrated and standalone software that both automate and improve the entire process.

Some background: When a roof is replaced through insurance, insurance will initially send out a payment that is only a fraction of the amount that insurance is covering for the full roof replacement. Once the roofing company has completed the elected work for the homeowner, they will send proof of completion off to the insurance company and ask that the remainder of the claim be paid out to the homeowner, making the homeowner full on the claim.

Where this task can become tedious: The roofing company may not complete all line items found within the insurance paperwork. The roofing company may forego entire trades or it might be the case that an individual line item, like chimney flashing, for example, was not completed. So for each of the thousands of roofs that are worked on each year, the company needs to ensure that all invoicing on both the homeowner and insurance side matches the actual line items completed.

The existing workflow: The invoicing department receives a PDF copy of the insurance paperwork which itemizes all work that the insurance company is willing to cover for their policyholder, aka the homeowner. The employee in the invoicing department of the roofing company then uses notes and orders within the company's business management system to determine which of the insurance line items were completed throughout the course of the roofing job.
Where this process typically is less straightforward than it sounds: The differing trades (roof, gutters, siding) are all completed at different times, homeowners may change their minds on the product and what work to have completed during the roofing process, there may be portions of work that were not originally noticed by the insurance company and supplemental work is later added to the claim, etc. Because the paperwork exists in PDF, the employee then manually tallies each of the completed line items to each of the trade groups completed. An example of this might be:

Remove/Replace Roof
(Line Items #1-20, 24, 27)
$18,500
Detach/Reset Gutters & Downspouts
(Line Items #28, 29)
$3,250
Remove/Replace Siding
(Line Items #34-37, 42-45)
$12,400
Remove/Replace Fascia
(Left Elevation Only - Line Item #52)
$1,800


Again, it is typical that not every line item within the insurance paperwork gets billed for. The cherrypicking of doing this line item here, but not this one here, etc, etc creates additional work in that the numbers must be reconciled to ensure no items were skipped. This results in the employee spending much more time than necessary for what should be a relatively easy and straightforward task.

The new workflow: The software that I created is an interface that exposes the user to all of the following:

  • Parsing insurance paperwork pdfs to csv / table format
  • The ability to group table items by trade (roof, gutters, etc)
  • The ability to make new trade groups
  • Dollar amount totals for each of the trade groups
  • Ability to total by other numeric amounts (Depreciation, ACV, Tax, etc)
  • A list of line items included in each of the trade groups
  • An auto-reconciliation total to assert all items have been correctly parsed and accounted for
  • Export functionality to setup the data for the next part of the invoicing process

Insurance Formats

Multiline Formats

Additionally, insurance paperwork from the same insurer might have differing formats. The most variation in parsing comes from inline vs. multiline line items. Inline is what it sounds like; all line item datapoints fall on the same line horizontally. Whereas multiline items might include several lines (vertically) for a single line item.

Both algorithms search for a regular expression to find the item description (i.e. 12. R&R Chimney Flashing). The inline algorithm then looks for all elements that occur to the right of the item description. The multiline algorithm will draw a bounding box which is separately configured for each of the insurance companies to find any overlapping elements that occur within that bounding box. If it is multiline paperwork, the user simply checks the Multiline Items? checkbox.

Column Sorter

The final step in parsing the PDF document is a column sorter dialog. This allows the user to assert that the parsed columns match up with the order of columns on the paperwork table. If it's necessary to change the order of the column headers for any reason, this dialog exposes drag and drop functionality to allow the user to rearrange the columns.

Background: As a sales contractor progresses through a program of internally tracked milestones--like the number of completed roofing jobs or the size of their sales team for example--the contractor will graduate to higher payout tiers that entail a better, albeit more complicated, compensation structure.

Payables

The user then enters the gross profit for the job and whether that job qualifies for the compensation program. If the job meets requirements for the program, the application will compute the compensation payouts attributable to the lead sales contractor and their team.

New Job Processing

Team Relationships

Team Overview / Org Chart

The purpose of this tab is to provide an overview of a sales contractor and their team--it serves as a quickview for specific individual & team stats as well as the hierarchy of each team member within their team.

Individual Performance Stats

About

This was a commissioned app that listens to a user-provided Twitter (Now X) handle that provides NBA player updates. The app in theory aims to take advantage of the discrepancy between new, live info about a player's status crossing and the time it takes for that information to be reflected in game/betting lines. If a discrepency exists, the app automatically places a bet looking to take advantage of this discrepancy.

Details: The user would configure which Twitter handle they wanted to listen to for player updates and then connect to the Twitter API. Then the user would set an additional two tables in the app configuration. One was a 'trigger' table. It is a series of player names and a respective list of actionable verbs or statuses (these could be set to use a configured set of default verbs). The second table was a 'bet' table in which the user would configure either 'team' or 'player' bets and then link those bets to triggers in the 'trigger' table (player + verb). If a new tweet crossed containing both the player name and any of the configured verbage, the bets linked to that specific trigger would be placed on the bookmaker's site. This was an entirely automated workflow, aka the end-user would just setup configuration variables, start it, and let it run.

What I Used:

  • Requests for streaming new tweets from the Twitter API
  • Selenium for interacting with web elements of the bookmaker site
  • PyQt for the user interface

What I Learned:

I thought that there would be 0% chance that discrepencies existed, or basically that the tweeted info would have already been reflected in the lines by the time the app received the update. Through basic testing it actually looked like these opportunities existed for a substantial amount of time, something like upwards of 30 seconds. A.k.a. plenty of time to take advantage of programatically...

About

This was a commissioned piece of software. Personal Seat Licenses are licenses that are bought and sold on marketplaces, and which entitle the licensee to purchase season tickets for NFL teams.

Why this software exists: for any given team, there may be something like 500 licenses available for sale at any given point in time. My client wanted a tool that would easily differentiate any new listings from existing listings, & the client also wanted the added ability to be notified of listings in realtime and the ability to further filter any new license listings to specified criteria (e.g. meeting a specific price range for a particular section).

PSL Marketplace Example

Here's an example of the license marketplace for the Minnesota Vikings.

Team Configuration

The user sets the configuration file for each NFL team they'd like to track PSLs for. The configuration file maps each team to a marketplace URL. Under the hood, the app will create a headless browser, and for each of the configured URL mappings, the browser will visit each of those websites. The app will then find current license listings and compare those to the ones already existing in the database for that specific team. If new license listings are found and they meet any further specified or configured criteria, the end user receives a message through Telegram.

  • pythonanywhere.com for hosting the code & database and for task scheduling
  • selenium for running a browser and interacting with web elements
  • telegram-python the python telegram client for creating and running a bot server

What I Learned:

Although a simple app, I had most difficulty with the loading of marketplace webpages. For some reason they are just really slow web pages (maybe because they are .aspx). This resulted in additional code that would either have to assert a successful marketplace load, or in the event of an unsuccessful load, that all of the existing database entries for that specific team's tickets were not deleted from the database --> basically a page would not load, the app would read this as that team not having any tickets for sale (no listing elements found), and then the app would delete all existing database entries because it assumed that all of those listings were delisted. On the next update and successful page load, the app would then send like 500 messages for each listing thinking that all of these listings were new (empty database table).

I also had to determine the most efficient task schedule in order to not exceed the amount of cloud resources aka compute time that the client was willing to pay for. This tool was also a good refresher on Telegram bot messaging and error catching / waits in selenium.
