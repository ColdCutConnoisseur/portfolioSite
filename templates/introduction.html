Desktop App

This app handles all authentication on the broker side (and functionality that continues to refresh authentication as tokens near expiration). It then subscribes to market streams for any equities set to trade in the app's configuration. I use a python requests stream / session object to handle market update messages and keep a current orderbook for anything I am trading. A separate thread is started and a timeout occurs every (configured) few seconds. On a timeout, the application will poll the flask app to see if any new trade signals have arrived since the last poll. If a new signal has occurred, the app translates and formats a new order to my broker's API. An audit trail is provided both in the 'Processed Signals' and the 'Trade Review' windows. My trading exposure is then updated in the 'Open Positions' window.

About

This was software commissioned for a roofing company in Illinois to better track their internal compensation structure for members of its sales team.

As a sales contractor progresses through a program of internally tracked milestones--like the number of completed roofing sales and jobs or the size of their sales team for example--the contractor will graduate to higher payout tiers that entail a better, albeit more complicated, compensation structure.

Overview

This is an overview of the 4-tabbed interface. On this primary page, a dropdown list is populated with unprocessed jobs that were closed during the current week. It also populates the current compensation stats for the sales contractor associated with the currently selected job.

Unprocessed Jobs

The jobs are sorted by the 'close date' to ensure that they are processed in order. When a job is selected from the dropdown, all associated information for the related sales person is automatically displayed.

Payables

The user then enters the gross profit for the job and whether that job qualifies for the compensation program. If the job meets requirements for the program, the application will compute the compensation attributable to the lead sales contractor and their team.

New Job Processing

After confirming the compensation amounts, the user will click the 'Mark Job as Processed' button, which updates all attributable job and compensation information within the company's database.

Team Relationships

This page is responsible for creating or updating the relationships between a sales contractor and their team.

Team Overview / Org Chart

The purpose of this tab is to provide an overview of a sales contractor and their team--it serves as a quickview for specific individual & team stats as well as the hierarchy of each team member.

Individual Performance Stats

The third tab within the application helps track additional, peripheral attributes for sales contractors. It houses attributes such as the selected sales person's beginning job count, whether or not that contractor has any direct hires themselves, and also allows the user to overwrite the compensation tier that the contractor starts at.

This functionality allows for this software to be employed in the midst of a sales year or for employees that have negotiated different starting tiers based on professional experience.

Tier Overwrite Functionality

The compensation tier overwrite allows functionality to start sales contractors at a compensation tier that exceeds whatever their natural compensation tier would be, given that contractors number of completed jobs, etc, at the beginning of tracking this program. It also allows for the sales person to graduate to further compensation tiers once the requirements for those higher tiers have been met organically.

About

This app listens to a user-provided Twitter handle that provides NBA player updates. The app in theory aims to take advantage of the discrepency between new info about a player's status crossing and the time it takes for the information to be reflected in game lines. If a discrepency exists, the app automatically places a bet.

Details: The user would configure which Twitter handle they wanted to listen to for player updates and then connect to the Twitter API. Then they would set an additional two tables in the app configuration. One was a 'trigger' table. It is a series of player names and a respective list of actionable verbs or statuses (these could be set to use a configured set of default verbs). The second table was a 'bet' table in which the user would configure either 'team' or 'player' bets and then link those bets to triggers in the 'trigger' table (player + verb). If a new tweet crossed containing both the player name and any of the configured verbage, the bets linked to that specific trigger would be placed on the bookmaker's site. This was an entirely automated workflow, aka the end-user would just setup configuration variables, start it, and let it run.

What I Used:

  • Requests for streaming new tweets from the Twitter API
  • Selenium for interacting with web elements of the bookmaker site
  • PyQt for the user interface

What I Learned:

I thought that there would be 0% chance that discrepencies existed, or basically that the tweeted info would have already been reflected in the lines by the time the app received the update. Through basic testing it actually looked like these opportunities existed for a substantial amount of time.
